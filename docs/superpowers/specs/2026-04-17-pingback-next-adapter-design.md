# @pingback/next Adapter — Design Spec

**Date:** 2026-04-17
**Scope:** Next.js adapter package for Pingback — build plugin, route handler, function registration

## Overview

`@pingback/next` is a thin Next.js adapter over `@pingback/core`. It lets developers define cron jobs and background tasks in their codebase, auto-generates a route handler, and registers functions with the Pingback platform at build time.

## Decisions

- **Build plugin approach:** `withPingback()` config wrapper (not a raw webpack plugin) — standard Next.js pattern, works with webpack and Turbopack
- **Route handler:** Code-generated at `app/api/__pingback/route.ts` during build — gitignored, standard App Router route
- **Registration timing:** During `next build` as part of the `withPingback()` pre-build step
- **Config:** Minimal — only `apiKey` required, everything else inferred or optional
- **Error handling:** Handler errors return HTTP 500 so the worker retries. Success returns 200.

## Developer Experience

### 1. Install

```bash
npm install @pingback/next
```

### 2. Create config — `pingback.config.ts`

```ts
import { defineConfig } from "@pingback/next";

export default defineConfig({
  apiKey: process.env.PINGBACK_API_KEY,
});
```

### 3. Wrap Next.js config — `next.config.ts`

```ts
import { withPingback } from "@pingback/next";

export default withPingback({
  // normal next.js config
});
```

### 4. Define functions

```ts
// lib/pingback/review-emails.ts
import { cron } from "@pingback/next";

export const sendReviewEmails = cron(
  "send-review-emails",
  "*/15 * * * *",
  async (ctx) => {
    const pending = await getPendingEmails();
    ctx.log(`Processing ${pending.length} emails`);
    return { processed: pending.length };
  },
  { retries: 3, timeout: "60s" }
);
```

### 5. Set env vars

```
PINGBACK_API_KEY=pb_live_...       # For build-time registration
PINGBACK_CRON_SECRET=...           # From project settings, for runtime HMAC verification
```

### 6. Build & deploy

```bash
next build
```

Everything is automatic — function discovery, route generation, platform registration.

## Component Details

### `functions.ts` — `cron()` and `task()`

Thin wrappers that delegate to core's `Registry`:

```ts
import { Registry } from "@pingback/core";

const registry = new Registry();

export function cron(name, schedule, handler, options?) {
  return registry.cron(name, schedule, handler, options);
}

export function task(name, handler, options?) {
  return registry.task(name, handler, options);
}

export { registry };
```

The shared `registry` instance is the single source of truth. Importing a function file triggers the `cron()`/`task()` call, which self-registers.

### `config.ts` — `defineConfig()`

```ts
interface PingbackConfig {
  apiKey: string;          // Required — PINGBACK_API_KEY
  baseUrl?: string;        // App's public URL (default: VERCEL_URL or localhost:3000)
  platformUrl?: string;    // Pingback API URL (default: https://api.pingback.dev)
  routePath?: string;      // Generated route path (default: /api/__pingback)
  functionsDir?: string;   // Glob for function files (default: "lib/pingback/**/*.{ts,js}")
}
```

`defineConfig()` returns the config object with defaults applied. At build time, the config file is loaded via dynamic import.

Config resolution order:
1. Look for `pingback.config.ts` at project root
2. Fall back to `pingback.config.js`
3. Load via `tsx` (same mechanism used for evaluating function files)
4. Merge with defaults

### `plugin.ts` — `withPingback(nextConfig)`

Returns a modified Next.js config. During build, it:

**Step 1 — Load config:**
- Reads `pingback.config.ts` from the project root via dynamic import

**Step 2 — Discover functions:**
- Globs for files matching `functionsDir` pattern (default: `lib/pingback/**/*.{ts,js}`)
- Filters to files that import from `@pingback/next`

**Step 3 — Generate route handler:**
- Creates `app/api/__pingback/route.ts` (or custom `routePath`):
  ```ts
  // AUTO-GENERATED BY @pingback/next — do not edit
  import { createRouteHandler } from "@pingback/next/handler";
  import "../../lib/pingback/review-emails";
  import "../../lib/pingback/data-sync";

  export const POST = createRouteHandler();
  ```
- Each discovered file gets an import line — importing triggers `cron()`/`task()` registration
- If `app/api/__pingback` is not in `.gitignore`, logs a warning

**Step 4 — Register with platform:**
- Evaluates the discovered files to populate the registry (using `tsx` or `ts-node`)
- Calls `registry.getMetadata()` to collect function definitions
- Calls core's `RegistrationClient.register(projectId, functions)` with the API key
- Project ID: the registration endpoint resolves it from the API key (the platform knows which project the key belongs to, so we don't send project_id — the platform infers it)

**Step 5 — Return config:**
- Returns the original `nextConfig` unmodified (the route file is already generated as a static file)

### `handler.ts` — `createRouteHandler()`

Returns a Next.js App Router `POST` handler function:

**1. Verify request:**
- Extract `X-Pingback-Signature` and `X-Pingback-Timestamp` from headers
- Read raw request body as text
- Load `PINGBACK_CRON_SECRET` from `process.env`
- Call core's `verifySignature(bodyObject, signature, parseInt(timestamp), secret)`
- If invalid → return `Response` with 401

**2. Dispatch:**
- Parse body: `{ function, executionId, attempt, scheduledAt }`
- Look up function by name via `registry.get(name)`
- If not found → return `Response` with 404
- Create context via core's `createContext(payload)`
- Start timer, execute `await definition.handler(ctx)`

**3. Respond:**
- On success → return `Response` with 200:
  ```json
  {
    "status": "success",
    "result": <return value>,
    "logs": <ctx._getLogs()>,
    "durationMs": <elapsed>
  }
  ```
- On handler error → return `Response` with 500:
  ```json
  {
    "status": "error",
    "error": <error.message>,
    "logs": <ctx._getLogs()>,
    "durationMs": <elapsed>
  }
  ```
  500 triggers the worker's retry logic.

### `register.ts` — Build-time registration

Standalone module used by the plugin:

```ts
async function registerFunctions(config: PingbackConfig, functionFiles: string[]) {
  // 1. Evaluate function files to populate registry
  for (const file of functionFiles) {
    await import(file);
  }
  
  // 2. Collect metadata
  const metadata = registry.getMetadata();
  
  // 3. Determine base URL for the endpoint
  const baseUrl = config.baseUrl 
    || process.env.VERCEL_URL && `https://${process.env.VERCEL_URL}`
    || 'http://localhost:3000';
  const endpointUrl = `${baseUrl}${config.routePath || '/api/__pingback'}`;
  
  // 4. Call registration API
  const client = new RegistrationClient(
    config.platformUrl || 'https://api.pingback.dev',
    config.apiKey,
  );
  // Note: project_id is inferred from API key on the platform side
  // We need to update the registration endpoint or client to support this
  return client.register(projectId, metadata);
}
```

**Note on project_id:** The current `RegistrationClient.register()` requires a `projectId` parameter. Since we want the API key to be the only required config, the platform's `/api/v1/register` endpoint should accept requests without `project_id` and infer it from the API key. This requires a small platform-side change: make `project_id` optional in `SdkRegisterDto`, and if omitted, use the project from the authenticated API key.

## Platform Changes Required

### Make `project_id` optional in registration

In `apps/platform/src/modules/projects/dto/register.dto.ts`:
- Change `project_id` from `@IsUUID()` required to `@IsUUID() @IsOptional()`

In `apps/platform/src/modules/projects/registration.controller.ts`:
- If `dto.project_id` is provided, validate it matches the API key's project (existing behavior)
- If omitted, use `project.id` from the authenticated API key

In `packages/core/src/registration.ts`:
- Make `projectId` optional in `register()` method
- If not provided, omit `project_id` from the request body

## Dependencies

`@pingback/next` depends on:
- `@pingback/core` — registry, context, signing, registration client
- `glob` — file discovery at build time
- `tsx` — evaluating TypeScript function files at build time

Peer dependencies:
- `next` >= 14

## Testing Strategy

- **`functions.ts`**: Test that `cron()` and `task()` register into the shared registry
- **`handler.ts`**: Test HMAC verification, function dispatch, success/error responses (mock registry and signing)
- **`config.ts`**: Test defaults, merging, validation
- **`plugin.ts`**: Test file discovery, route generation, registration call (integration-level with mocks)
- **`register.ts`**: Test that files are evaluated and registration client is called with correct metadata
